Small Features/Bugs  – 1 Point Stories
Review by a pair should be sufficient for making sure the code meets business and technical acceptance criteria

Large Features – 2,3 Point Stories
Should be pull requested on a separate branch and approved by someone outside the assigned pair.

Acceptance Criteria for Reviewing a PR
* Formatting - PRs should follow template (reference Common Engineering Policy) & be filled out by requester.
* Functionality - Pull code and ensure features in the story work properly.
* Clean code - Ensure coding standards were upheld with the newly introduced code.
* Tests - Make sure all tests pass and proper test were written to test new code.
* Cupcake emojis 🧁  - Comments that include a cupcake should be used as suggestions in the PR comments. These comments are meant to be recommendations by the reviewer and should not hold up merging of a PR.
