## Backlog Hierarchy (Features, Sub-Epics, Epics)

 Item      | Hierarchy     | Duration     | Description                                                                                                                                                                                                           |
|-----------|---------------|--------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Epic      | Product Group | > 6 months   | A large development initiative aligned with the strategic intent of the business.                                                                                                                                     |
| Sub-Epic  | Product Line  | 3-6 months   | Major capability that can be understood and communicated across all of Platform+. Taken all together, the sub-epics tell an easily consumable story of all work from all teams.                                        |
| Feature   | Product Team  | < 3 Months   | Functions that fulfill a user need.  A new or improved capability of the product.  Features deliver a package of functionality.                                                                                       |

Note: The above hierarchy and duration is a guideline and not a hard rule. As always, use professional judgement to
structure a backlog that serves your team well.

## Backlog Hierarchy Refinement

![Backlog Items](/images/pm/backlog_items.svg)

* Top-Down – An Epic is defined aligned to an organizational initiative then further research is done to decompose it
  into Sub-Epics, Features and Stories
* Bottom-Up – A feature is delivered to provide user value. As the team learns, it discovers the need for additional
  features which may not always be directly tied to a Sub-Epic or Epic in the backlog
* While PDO material typically refers to top-down discovery, both types of discovery are encouraged within Platform+.
  Product Group cascades down strategic objectives and the product teams cascade up learnings.

  ## Writing Features, Sub-Epics, & Epics
  There are three common ways to author a backlog item. Choose the template that you feel will best explain to the team
why they are undergoing this development effort. Additionally, you can use your own format as long as it brings the team
a common understanding of the portfolio item.

| Title                  | Description                                                                                                                | When to Use                                                                                                                                 |
|------------------------|----------------------------------------------------------------------------------------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------|
| Hypothesis Driven      | Backlog item written in a form of a hypothesis                                                                             | When there are large degrees of uncertainty and learning is a key objective                                                                 |
| What/Why/Done          | Plain backlog item clearly spelling out what needs to be done, why it needs to be done and what is required for acceptance | When capabilities need to be added to a system that may not directly be requested by a user (persona).  Back end processes are a great fit. |
| As a… I want… So that… | Traditional "story" format that explains the backlog item from the perspective of a user (persona)                         | When the backlog item will provide demonstrable value to the user (persona)                                                                 |

### Hypothesis Driven

This template is best to use when there is uncertainty in the effectiveness of the proposed solution. The backlog item is stated in the form of a hypothesis which ultimately encourages the team to reflect on the outcomes the backlog item is achieving and change course if the hypothesis is invalidated (or drop the item altogether). Because this template is used to address uncertainty it is best used for Epics, Sub-Epics and occasionally features.

- Hypothesis
   - We believe that:• making this [the change]• for [target users]• will achieve [this impact]

- Leading & Lagging Indicators
   - How will we know this feature is successful?
      - We will know this is true when we see • [measurable signal - likely your leading indicators] • in [timeframe - when can we check our leading indicators]
    - Leading indicator: list here
    - Lagging indicator: list here

- Acceptance Criteria
   - How will we know the feature is done?

- Risks & Dependencies
   - What risks and dependencies will we encounter when delivering this feature?
   - Risk should be in IF/THEN Format: If [Event occurs] Then [Impact to the feature]

### What/Why/Done
Common with User Story Writing, this format can also be used in feature writing.  Follow the same format in [User Stories here](/docs/practices/product-management/techniques-tools/product-development-delivery/backlog-artifacts#whatwhydone-additional-user-story-format)

### As a... I want... So that...
Common with User Story Writing, this format can also be used in feature writing.  Follow the same format in [User Stories here](/docs/practices/product-management/techniques-tools/product-development-delivery/backlog-artifacts#business-value-format)

### Acceptance Criteria

Scenario [Title]

- Acceptance criteria are testable statements with clear pass/fail results.
- They should state intent, not a specific solution.
- Consider functional, non-functional, and performance criteria.

Repeat for each scenario that needs to be tested

#### Exclusions

Is there anything relevant to this work that should not be included in the completion of this work?

#### Other Notes

Great way to pass along additional context to the team.

- Link to Miro Board
- Link to Design Mockups
- Technical Implementation Notes
- Key Contacts
